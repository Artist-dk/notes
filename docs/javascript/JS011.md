# ðŸ“Œ **7. Handling Errors in Callbacks**
A **common pattern** in callback functions is the use of **error-first callbacks**.

### **ðŸ”¹ Error-First Callback Example**
```js

            callback("Invalid ID", null); // Pass error message
        } else {
            callback(null, { id: id, name: "Alice" }); // Pass user data
        }
    }, 2000);
}

    if (error) {
        console.error("Error:", error);
    } else {
        console.log("User Data:", user);
    }
});
```
âœ” **Output:**
```

```
ðŸ”¹ The **first parameter** in the callback is reserved for **errors**, while the second contains the actual data.

# ðŸ“Œ **8. Callback Hell (Nested Callbacks Problem)**
When multiple callbacks are **nested**, code becomes **difficult to read** and **hard to maintain**. This problem is called **Callback Hell**.

### **ðŸ”¹ Example: Nested Callbacks (Callback Hell)**

        console.log("Step 3 complete");
    }, 1000);

// Nested Callbacks (Callback Hell)

    step2(() => {
        step3();

ðŸ”¹ This structure is **hard to manage** for larger projects.

ðŸ”¹ **Solution:** Use **Promises** or **Async/Await** to improve readability.
